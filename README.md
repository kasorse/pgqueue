## Концепция

pgqueue позволяет обрабатывать задачи, для хранения которых используется
таблица `queue`, которая заранее создается владельцами сервиса. Задача может
содержать произвольный пейлоад в формате JSON. Ошибки обработки ретраятся
в течение сконфигурированного числа попыток.

Пользователь создает в main своего сервиса **процессор задач** и для каждого вида
задачи регистрирует **обработчик** — структуру с методом, принимающим задачу и
возвращающим ошибку. Затем пользователь планирует задачи в транзакции. Процессор
задач в цикле выбирает задачи из таблицы, переводит их в статус «обрабатывается»
и отдает задачи в пул **воркеров**, запускающих зарегистрированные пользовательские
обработчики. Успешно выполненная задача помечается в таблице соответствующим статусом.

pgqueue дает гарантию at-least-once, однако не дает гарантии exactly once, и к
тому же возможны ситуации, когда одна задача будет выполняться конкурентно двумя
воркерами (см. [раздел про обработку потерянных задач](#обработка-потерянных-задач)).

## Как использовать

1. Скопировать в свой сервис миграции из папки [migrations](migrations) и накатить их.
2. Реализовать сборщика метрик, имеющего интерфейс `pgqueue.MetricsCollector`.
3. Реализовать обработчика тасок, имеющего интерфейс `pgqueue.TaskHandler`.
4. В main своего сервиса запустить процессор задач.
5. Планировать таски с помощью метода `AppendTask`, если нужно планировать
в транзакции — с помощью метода `SQLAppendTask`.

См. игрушечный пример в папке [example](example).

### Важные настройки

При создании процессора рекомендуется обратить внимание на следующие настройки:
1. `MaxAttempts` — число попыток обработки задачи. По умолчанию 1, то есть
после первой ошибки задача завершится и ретраиться не будет. Нужно ставить
самое большое число, которое разумно для данного вида задач.
2. `AttemptLimitSeconds` — таймаут выполнения задачи. Если задача выполняется
дольше, она вернется в очередь, и ее может начать выполнять другой воркер.
Поэтому таймаут должен быть достаточно большим, чтобы ожидаемое время выполнения
задачи помещалось в него. В то же время, если таймаут будет очень большим, то
увеличится время, которое потребуется для выполнения задачи, попавшей в
процессор задач на умершем поде.
3. `DelayAfterRefusedSeconds` / `ProgressiveDelayAfterRefusedSeconds` —
задержка перед очередной попыткой выполнения. Подробнее см. в
[разделе про обработку ошибок](#обработка-ошибок).

## Обработка ошибок

Ошибки пользовательского обработчика ретраятся в течение `MaxAttempts` раз
(по умолчанию 1). Если попытки закончились, задача завершается со статусом
99 (`ClosedNoAttemptsLeft`). C помощью метода `AddRetries` можно добавить
попыток всем таким задачам для выбранного вида задач.

Задержка перед очередной попыткой обработки задается опциями
`DelayAfterRefusedSeconds` и `ProgressiveDelayAfterRefusedSeconds`.
Опция `DelayAfterRefusedSeconds` задает постоянную задержку в секундах,
опция `ProgressiveDelayAfterRefusedSeconds` задает последовательность
задержек в секундах и имеет приоритет над `DelayAfterRefusedSeconds`.

## Обработка потерянных задач

Возможна ситуация, когда процессор задач переместил задачу в статус
«обрабатывается», но после этого умер или завис. Для обработки этого
случая на одну итерацию обработки задачи накладывается таймаут
`AttemptLimitSeconds`. Процессор задач в отдельной горутине периодически
находит задачи, которые выполняются дольше лимита, и возвращает их в очередь.
Таким образом, если процессор начал выполнять задачу и сразу после этого умер
под, то процессор на каком-либо другом поде через `AttemptLimitSeconds` секунд
вернет задачу в очередь, после чего ее подберет любой живой процессор.

## TTL задач

Если задача не завершилась в течение `TTLSeconds` секунд после создания, то она
переводится в статус 98 (`ClosedExpired`). C помощью метода `AddRetries` можно
сбросить число попыток всем таким задачам для выбранного вида задач, после чего
они вернутся в очередь.

Если опция `TTLSeconds` не задана, то она вычисляется как число секунд, которое
потребуется на все `MaxAttempts` попыток с учетом настроек `AttemptLimitSeconds`
и `DelayAfterRefusedSeconds` / `ProgressiveDelayAfterRefusedSeconds`.
